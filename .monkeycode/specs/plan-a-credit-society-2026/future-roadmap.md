# 方案 A 后续路线图

## 1. 文档目的

本文件用于描述方案 A 在首轮落地后的持续开发方向，避免后续迭代重新回到功能堆砌路线。

与其他文档的关系如下：

- `requirements.md`：定义方案 A 当前需要实现什么
- `design.md`：定义方案 A 如何设计
- `tasklist.md`：定义当前可执行的实现清单
- `implementation-order.md`：定义最少返工的落地顺序
- 本文件：定义首轮落地后，还应沿什么方向继续扩展

## 2. 首轮落地后的三个阶段

### 阶段 A：把画像从“有规则”做成“有存在感”

重点：

- 增强画像展示频率
- 让周结算、日志、借贷弹窗更强地表达画像变化
- 让玩家能复述自己的当前制度身份

成功标志：

- 玩家能说出自己目前被系统看作什么人
- 玩家能感知某些事件和待遇变化不是随机，而是画像在起作用

### 阶段 B：把身体抵押做成第二主轴

重点：

- 扩展身体部位抵押的收益和代价种类
- 让身体状态长期影响事件池和制度标签
- 将身体资产折损和画像变化形成更强联动

成功标志：

- 玩家会主动记住自己“卖掉了什么”
- 玩家后续遇到的世界反馈明显受身体状态影响

### 阶段 C：把多周目做成命运实验室

重点：

- 多周目继承的不只是数值，而是命运档案与社会偏见
- 某些事件和待遇因上一局经历而不同
- 系统逐渐“记得”老玩家的倾向

成功标志：

- 新一局不再只是重开，而是进入一个更理解你的世界
- 玩家会讨论“上一世留下了什么后果”

## 3. 后续功能池的优先级

### P1：应优先持续推进

- 画像标签扩展与调优
- 画像驱动的 NPC 和事件反馈
- 身体资产市场化
- 制度归档和命运分享卡
- 多周目命运档案

### P2：可以在主轴稳定后接入

- 反画像路线
- 误判与纠偏事件
- 更细粒度的宗门、老师、同学画像反馈
- 更多制度报告样式

### P3：主轴完成前不建议优先

- 常规签到
- 传统成就体系
- 与主题弱耦合的通用扩展功能
- 纯视觉先行的大规模重构

## 4. 后续版本建议

### Version Next

目标：

- 画像系统站稳
- 周结算和借贷反馈完成画像化
- SummaryPanel 完成制度归档升级

### Version Next+1

目标：

- 身体抵押成为半主系统
- 事件池开始明显按画像差异化
- FateCard 或命运判定卡完成首版

### Version Next+2

目标：

- 引入反画像路线的首版
- 增强误判、纠偏和画像追踪升级
- 形成更完整的制度张力

### Version Long-term

目标：

- 多周目命运档案
- 世界对老玩家行为模式的继承性反馈
- 将项目从单局制度压迫，扩展成跨周目的命运实验

## 5. 后续迭代的判断标准

每次规划新版本时，都建议用以下标准筛选：

- 是否让“被定义感”更强
- 是否让世界反馈更像按画像运行
- 是否让身体、债务、分班、契约和精神系统更统一
- 是否让项目更容易被一眼记住

如果不满足这些标准，说明该功能更适合作为次优先级，而不是主线推进内容。
